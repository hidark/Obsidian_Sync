## 专题 A：打赏 100% vs 订阅 A/B 验证设计

C3.6 已定论"稀释 vs 筛选取决于 C1.5 守住"，但需要可**实际测量**的数据口径和实验设计，让 PRD 能写入监控机制。

### A.1 验证指标定义
列出 5-7 个可监控的数据口径，用于运行期判定"当前处于稀释还是筛选模式"：
- 订阅者打赏 lift = 订阅者人均打赏 / 非订阅者人均打赏（Twitch 参照 3-5x）
- 订阅-打赏时间序列协方差
- 创作者打赏 CTA vs 订阅 CTA 频次（需埋点）
- 其他...

### A.2 A/B 实验设计
PRD 上线后如何做 A/B 验证？给出实验组/对照组设计：
- 变量 1：订阅创作者功能 on/off（对创作者池）
- 变量 2：打赏 CTA 置于订阅入口旁 vs 远离
- 变量 3：订阅者打赏场景化（节日/章节高潮） vs 普通打赏按钮
- 样本量要求 / 实验期 / 停止规则

### A.3 护栏机制
如果运行期监测到"已进入稀释模式"，PRD 应有哪些应急方案？
- 产品机制（不是分成调整，因为 C1.7 归属链路未具备）
- 警戒阈值触发什么行动

### A.4 打赏场景化产品建议
C3.6 指出"打赏场景化是筛选成立的关键"。给出 3-5 个可加入 PRD 的打赏场景化产品：
- 节日打赏（母亲节/生日/角色纪念日）
- 章节高潮点打赏（订阅内容中触发）
- 订阅者专属打赏礼物
- 互动彩蛋（打赏触发特殊回执）

## 专题 B：NSFW 与 SFW 同步上线的支付过渡方案

用户要求不延后 NSFW。CCBill/Segpay 批复周期 2-6 周，假设是"发布日前必须已批下"。PRD 需给发布前过渡方案 + 发布后并行运营方案。

### B.1 发布前依赖清单（按 C1.6 BLOCKER）
列出发布前必须完成的支付/合规准备：
- CCBill/Segpay 主体申请（时间、所需材料）
- 2257 合规（每个 NSFW 创作者的 KYC 档案）
- VISA/Mastercard 高风险商户申请
- 退款策略、chargeback 响应话术、争议处理流程

按依赖关系排序，哪些是发布日硬门槛

### B.2 iOS 合规方案 A' 的具体落地（C2.3）
参考 Netflix/Spotify 做法：
- iOS app 内完全不提 NSFW 订阅入口
- Web 推送教育用户"订阅需在 Web 完成"
- 已订阅内容 iOS 内可查看
- 具体 UX 流程：用户在 iOS 发现一个 NSFW 创作者 → 看到什么 → 如何得知可在 Web 订阅 → 订阅后如何回 iOS 查看

### B.3 分成按扣渠道费后结算的执行细节
Phase 1 C1.3 定论：分成按扣渠道费后净收入结算。具体：
- 跨渠道（Web CCBill / Web Segpay / Web Stripe SFW 订阅 / iOS IAP / Android IAP）同一订阅价创作者收入不同，会计体验差
- 创作者看到的"订阅收入 tab"如何展示？混合展示 or 分渠道展示？
- 如果平台要"抹平"创作者体验，需平台补贴多少？是否可行？

### B.4 发布后 30/60/90 天监测指标
支付链路上线后的健康度监测：
- chargeback 率（目标 <1%，红线 >2%）
- 账户冻结预警
- 跨渠道对账稳定性
- 创作者提现成功率
